# 第十四章 准入控制完整生命周期——从请求到释放的全流程追踪

本章将整个准入控制系统串联起来，从一个请求的初始 `Admit()` 调用开始，经过排队、授权、执行，直到最终通过 `AdmittedWorkDone()` 清理资源的完整生命周期。我们将结合真实的源代码，提供每个关键点的具体示例。

---

## BFS Layer 1: 生命周期概览

### 1.1 核心阶段划分

一个请求的完整生命周期可以分为以下几个阶段：

```
┌─────────────────────────────────────────────────────────────────┐
│                     准入控制生命周期全景                          │
└─────────────────────────────────────────────────────────────────┘

阶段1: 准入请求（Admission Request）
    ↓
    调用 WorkQueue.Admit(ctx, info)
    ├─► 参数验证
    ├─► Bypass 检查（KVWork 特有）
    ├─► 快速路径尝试
    └─► 慢速路径排队

阶段2: 资源授权（Resource Granting）
    ↓
    GrantCoordinator.tryGrantLocked()
    ├─► 遍历 WorkKind 优先级
    ├─► 调用 Granter.tryGetLocked()
    ├─► Grant Chain 启动/继续
    └─► 通知等待的 goroutine

阶段3: 执行工作（Work Execution）
    ↓
    Goroutine 被唤醒，执行实际任务
    ├─► KVWork: 执行 Raft 提议
    ├─► SQLKVResponseWork: 处理 KV 响应
    └─► SQLSQLResponseWork: 处理 SQL 响应

阶段4: 资源释放（Resource Release）
    ↓
    调用 AdmittedWorkDone() / returnGrant()
    ├─► 归还 slots/tokens
    ├─► 更新租户统计
    ├─► 触发下一轮授权
    └─► Grant Chain 继续或终止
```

**请求信息流动**：

## BFS Layer 2: 三大典型场景

### 2.1 场景一：KVWork 快速路径（无排队）

**场景描述**：
- WorkKind: `KVWork`
- 队列状态：空
- Granter 状态：有可用 slots
- 结果：立即获得准入，无需排队

**流程简图**：
```
Request → Admit() → Fast Path → tryGet() → Success → Return
                       ↓
                   队列为空 + slots 可用
```

### 2.2 场景二：KVWork 慢速路径（需要排队）

**场景描述**：
- WorkKind: `KVWork`
- 队列状态：有等待请求
- Granter 状态：slots 耗尽
- 结果：进入队列等待，Grant Chain 授权

**流程简图**：
```
Request → Admit() → Slow Path → Enqueue → Wait
                       ↓                     ↓
                   快速路径失败          Block on channel
                                             ↓
                                      另一个请求归还 slot
                                             ↓
                                      Grant Chain 启动
                                             ↓
                                      Channel 收到通知
                                             ↓
                                      Goroutine 唤醒 → Execute
```

### 2.3 场景三：Store 层双重准入（写入请求）

**场景描述**：
- WorkKind: `KVWork`
- 类型：复制写入（Raft 日志）
- 需要：Store 层 IO tokens + CPU 层 slots
- 结果：两层都通过才能执行

**流程简图**：
```
Replicated Write Request
    ↓
    ┌─────────────────────┐
    │ 第一层：Store 层    │
    │ StoreWorkQueue      │
    │ ├─ IO tokens        │
    │ └─ L0 压力检查      │
    └─────────────────────┘
            ↓ (通过)
    ┌─────────────────────┐
    │ 第二层：CPU 层      │
    │ WorkQueue (KVWork)  │
    │ └─ CPU slots        │
    └─────────────────────┘
            ↓ (通过)
        Execute Raft Proposal
```

## DFS Layer 3: 准入请求阶段深度分析

### 3.1 Admit() 函数完整流程

**源代码位置**: `pkg/util/admission/work_queue.go:687-1089`

#### 3.1.1 六个处理阶段

```go
func (q *WorkQueue) Admit(ctx context.Context, info WorkInfo) (enabled bool, err error) {
    // ============================================================
    // 阶段 1: 准入控制启用检查
    // ============================================================
    tenantID := info.TenantID.ToUint64()

    q.mu.Lock()

    // 检查准入控制是否启用
    enabled = q.mu.admittingState.enabled()
    if !enabled {
        q.mu.Unlock()
        return false, nil  // ← 准入控制未启用，直接放行
    }

    // ============================================================
    // 阶段 2: 参数验证
    // ============================================================

    // 验证 requestedCount（slots/tokens 数量）
    if info.RequestedCount < 0 {
        q.mu.Unlock()
        return false, errors.AssertionFailedf("negative RequestedCount: %d", info.RequestedCount)
    }

    // KVWork 必须是 slot-based（count == 1）
    if q.workKind == KVWork && info.RequestedCount != 1 {
        q.mu.Unlock()
        return false, errors.AssertionFailedf("KVWork must request exactly 1 slot")
    }

    // ============================================================
    // 阶段 3: Bypass 处理（仅限 KVWork）
    // ============================================================

    // 获取或创建租户信息
    tenant, ok := q.mu.tenants[tenantID]
    if !ok {
        tenant = newTenantInfo(tenantID, q.getTenantWeightLocked(tenantID))
        q.mu.tenants[tenantID] = tenant
    }

    // KVWork 的 Bypass 机制
    if info.BypassAdmission {
        if q.workKind != KVWork {
            q.mu.Unlock()
            return false, errors.AssertionFailedf("BypassAdmission only for KVWork")
        }

        // 强制占用资源，不经过正常授权
        tenant.used += uint64(info.RequestedCount)
        if isInTenantHeap(tenant) {
            q.mu.tenantHeap.fix(tenant)
        }
        q.mu.Unlock()

        // 通知 granter 有资源被强制占用
        q.granter.tookWithoutPermission(info.RequestedCount)
        q.metrics.incAdmitted(info.Priority)
        q.metrics.recordBypassedAdmission(info.Priority)

        return true, nil  // ← 立即成功
    }

    // ============================================================
    // 阶段 4: 快速路径尝试
    // ============================================================

    // 更新优先级统计
    tenant.priorityStates.requestAtPriority(info.Priority)

    // 条件：队列为空 + 快速路径未禁用
    if len(q.mu.tenantHeap) == 0 && !q.knobs.DisableWorkQueueFastPath {
        // 乐观更新 used
        tenant.used += uint64(info.RequestedCount)
        q.mu.Unlock()

        // 尝试获取资源（无锁）
        if q.granter.tryGet(canBurst, info.RequestedCount) {
            // 成功！
            q.metrics.incAdmitted(info.Priority)
            q.metrics.recordFastPathAdmission(info.Priority)
            return true, nil  // ← 快速路径成功
        }

        // 失败，回退并进入慢速路径
        q.mu.Lock()
        tenant, ok = q.mu.tenants[tenantID]
        if !ok {
            tenant = newTenantInfo(tenantID, q.getTenantWeightLocked(tenantID))
            q.mu.tenants[tenantID] = tenant
        }
        if tenant.used >= uint64(info.RequestedCount) {
            tenant.used -= uint64(info.RequestedCount)
        } else {
            tenant.used = 0
        }
    }

    // ============================================================
    // 阶段 5: 慢速路径排队
    // ============================================================

    startTime := q.timeNow()

    // 检查上下文取消
    if ctx.Err() != nil {
        q.mu.Unlock()
        q.metrics.incErrored(info.Priority)
        return true, ctx.Err()
    }

    // 创建 waitingWork 对象
    work := getWaitingWork()
    *work = waitingWork{
        priority:       info.Priority,
        createTime:     info.CreateTime,
        requestedCount: info.RequestedCount,
        enqueueingTime: startTime,
        ch:             make(chan grantChainID, 1),  // ← 授权通知 channel
        replicated:     info.ReplicatedWorkInfo,
    }

    // 决定 FIFO 还是 LIFO
    useFIFO := tenant.priorityStates.getFIFOState(info.Priority)
    work.fifo = useFIFO

    if !useFIFO {
        // LIFO 模式：计算 epoch
        work.epoch = epochForTimeNanos(startTime.UnixNano(), q.epochLengthNanos)

        // 加入 openEpochsHeap（未关闭的 epoch）
        heap.Push(&tenant.openEpochsHeap, work)

        // 尝试关闭旧 epoch
        q.tryCloseEpoch(tenant, work.epoch)
    } else {
        // FIFO 模式：直接加入 waitingWorkHeap
        heap.Push(&tenant.waitingWorkHeap, work)
    }

    // 更新租户在 tenantHeap 中的位置
    if !isInTenantHeap(tenant) {
        heap.Push(&q.mu.tenantHeap, tenant)
    } else {
        q.mu.tenantHeap.fix(tenant)
    }

    q.mu.Unlock()

    // ============================================================
    // 阶段 6: 等待授权
    // ============================================================

    // 在 channel 上阻塞等待
    select {
    case <-ctx.Done():
        // 上下文取消
        q.cancelWaitingWork(tenant, work)
        q.metrics.incErrored(info.Priority)
        return true, ctx.Err()

    case chainID := <-work.ch:
        // 收到授权通知！
        q.mu.Lock()
        if work.heapIndex != -1 {
            // 还在 heap 中，移除它
            if work.inWaitingWorkHeap {
                tenant.waitingWorkHeap.remove(work)
            } else {
                tenant.openEpochsHeap.remove(work)
            }
        }
        releaseWaitingWork(work)  // 归还对象池
        q.mu.Unlock()

        // 记录等待时延
        waitDur := q.timeNow().Sub(startTime)
        q.metrics.recordFinishWait(info.Priority, waitDur)
        q.metrics.incAdmitted(info.Priority)

        // 继续 Grant Chain
        q.granter.continueGrantChain(chainID)

        return true, nil  // ← 慢速路径成功
    }
}
```

#### 3.1.2 快速路径 vs 慢速路径决策树

```
Admit() 调用
    ↓
准入控制启用？
    ├─► No → 返回 (false, nil)
    └─► Yes
        ↓
    参数有效？
        ├─► No → 返回错误
        └─► Yes
            ↓
        BypassAdmission = true？
            ├─► Yes → tookWithoutPermission() → 返回 (true, nil)
            └─► No
                ↓
            队列为空？
                ├─► No → 进入慢速路径（直接排队）
                └─► Yes → 尝试快速路径
                    ↓
                granter.tryGet() 成功？
                    ├─► Yes → 返回 (true, nil)  ✓ 快速路径成功
                    └─► No → 进入慢速路径
                        ↓
                    创建 waitingWork
                        ↓
                    FIFO or LIFO？
                        ├─► FIFO → push(waitingWorkHeap)
                        └─► LIFO → push(openEpochsHeap) + tryCloseEpoch
                        ↓
                    阻塞等待 channel
                        ↓
                    收到通知 → 返回 (true, nil)  ✓ 慢速路径成功
```

### 3.2 实例分析：KVWork 快速路径

**场景设定**：
- 一个 KVWork 请求到达
- 当前队列为空
- slotGranter 有可用 slots（totalSlots=100, usedSlots=50）

**代码执行路径**：

```go
// 步骤 1: 调用 Admit
info := WorkInfo{
    TenantID:       roachpb.MustMakeTenantID(2),
    Priority:       admissionpb.NormalPri,
    CreateTime:     timeNow(),
    RequestedCount: 1,  // KVWork 总是请求 1 slot
}

enabled, err := workQueue.Admit(ctx, info)

// 步骤 2: 快速路径条件满足
// 源代码: work_queue.go:826-867
if len(q.mu.tenantHeap) == 0 && !q.knobs.DisableWorkQueueFastPath {
    // 队列为空 ✓

    tenant.used += 1  // 乐观更新：50 → 51
    q.mu.Unlock()     // 释放锁

    // 步骤 3: 尝试获取 slot
    // 调用路径: granter.tryGet() → slotGranter.tryGet() → coord.tryGet()
    if q.granter.tryGet(canBurst, 1) {
        // slotGranter 检查
        // usedSlots=50 < totalSlots=100 ✓
        // usedSlots++ → 51

        // 成功！
        q.metrics.incAdmitted(info.Priority)
        q.metrics.recordFastPathAdmission(info.Priority)
        return true, nil  // ← 返回成功，无需等待
    }
}
```

**时间消耗**：
- 快速路径通常在 **1-10 微秒**内完成
- 无 channel 通信
- 无 goroutine 调度
- 仅一次 mutex 锁竞争

**Metrics 更新**：
```
admission.granter.slots_used{workKind=kv}:     50 → 51
admission.admitted.requests{workKind=kv, pri=normal-pri}:  +1
admission.admit.fast_path{workKind=kv, pri=normal-pri}:    +1
```

### 3.3 实例分析：KVWork 慢速路径

**场景设定**：
- 第 51 个 KVWork 请求到达（slots 已满）
- 队列中已有 10 个等待请求
- 使用 LIFO 模式，epoch 长度 100ms

**代码执行路径**：

```go
// 步骤 1: 快速路径失败（队列不为空）
// 源代码: work_queue.go:826
if len(q.mu.tenantHeap) == 0 && !q.knobs.DisableWorkQueueFastPath {
    // 条件不满足，跳过快速路径
}

// 步骤 2: 创建 waitingWork
// 源代码: work_queue.go:950-965
startTime := q.timeNow()  // 假设: 2026-01-08 10:00:00.250
work := getWaitingWork()
*work = waitingWork{
    priority:       admissionpb.NormalPri,
    createTime:     startTime.UnixNano(),
    requestedCount: 1,
    enqueueingTime: startTime,
    ch:             make(chan grantChainID, 1),  // 创建通知 channel
    fifo:           false,  // LIFO 模式
}

// 步骤 3: 计算 epoch
// 源代码: work_queue.go:1809-1811
epochLengthNanos := 100 * 1000000  // 100ms = 100,000,000 纳秒
work.epoch = startTime.UnixNano() / epochLengthNanos
// 例如: 1736326800250000000 / 100000000 = 17363268002

// 步骤 4: 加入 openEpochsHeap
// 源代码: work_queue.go:970-973
heap.Push(&tenant.openEpochsHeap, work)
// openEpochsHeap 现在包含这个请求

// 步骤 5: 尝试关闭旧 epoch
// 源代码: work_queue.go:440-540
q.tryCloseEpoch(tenant, work.epoch)
// 检查是否有 epoch < 17363268002 可以关闭

// 步骤 6: 更新 tenantHeap
// 源代码: work_queue.go:983-987
if !isInTenantHeap(tenant) {
    heap.Push(&q.mu.tenantHeap, tenant)
} else {
    q.mu.tenantHeap.fix(tenant)  // 重新调整位置
}

q.mu.Unlock()

// 步骤 7: 阻塞等待
// 源代码: work_queue.go:1062-1089
select {
case <-ctx.Done():
    // 上下文取消
    return true, ctx.Err()

case chainID := <-work.ch:
    // ← 阻塞在这里，等待 channel 通知
    // 可能等待 1ms ~ 数秒，取决于系统负载
}

// 步骤 8: 被唤醒（另一个 goroutine 调用了 granted()）
// 源代码: work_queue.go:1188
// item.ch <- grantChainID  ← 在 granted() 中发送

// 步骤 9: 清理并返回
q.mu.Lock()
if work.heapIndex != -1 {
    tenant.openEpochsHeap.remove(work)  // 从 heap 移除
}
releaseWaitingWork(work)  // 归还对象池
q.mu.Unlock()

// 步骤 10: 继续 Grant Chain
q.granter.continueGrantChain(chainID)

return true, nil  // ← 成功获得准入
```

**关键时间点**：
```

T1 = 10:00:00.251  - 加入 openEpochsHeap，开始等待
T2 = 10:00:00.350  - epoch 17363268002 关闭（100ms 后）
T3 = 10:00:00.351  - waitingWork 移动到 waitingWorkHeap
T4 = 10:00:00.420  - 另一个请求完成，归还 slot
T5 = 10:00:00.421  - Grant Chain 启动
T6 = 10:00:00.422  - granted() 发送 channel 通知
T7 = 10:00:00.423  - Goroutine 被调度器唤醒
T8 = 10:00:00.424  - continueGrantChain() 调用
T9 = 10:00:00.425  - 返回成功

总等待时间: T9 - T0 = 175ms
```

## DFS Layer 4: 资源授权阶段深度分析

### 4.1 GrantCoordinator 的授权循环

**源代码位置**: `pkg/util/admission/grant_coordinator.go:749-898`

#### 4.1.1 tryGrantLocked() 核心逻辑

```go
// tryGrantLocked 是授权的核心函数
// 它按照 WorkKind 优先级顺序，尝试为等待的请求授权
func (coord *GrantCoordinator) tryGrantLocked() {
    // ============================================================
    // 阶段 1: CPU 过载检查
    // ============================================================

    // 检查 CPU 是否过载
    if coord.mu.cpuOverloadIndicator.isOverloaded() {
        // CPU 过载，停止所有授权
        return
    }

    // ============================================================
    // 阶段 2: Grant Chain 状态检查
    // ============================================================

    startingChain := false
    if coord.useGrantChains && !coord.mu.grantChainActive {
        // Grant Chain 未激活，准备启动新的 Chain
        coord.mu.grantChainID++  // 生成新的 chain ID
        coord.mu.grantChainIndex = 0  // 从最高优先级开始
        startingChain = true
    }

    // ============================================================
    // 阶段 3: 多 WorkKind 授权循环
    // ============================================================

    // grantBurstLimit: 单次批量授权的最大请求数
    // 默认值: 2（减少 Grant Chain 长度，避免饥饿）
    grantBurstCount := 0
    grantBurstLimit := 2

OuterLoop:
    // 按优先级顺序遍历 WorkKind
    for ; coord.mu.grantChainIndex < numWorkKinds; coord.mu.grantChainIndex++ {
        workKind := WorkKind(coord.mu.grantChainIndex)
        granter := coord.granters[workKind]
        requester := coord.queues[workKind]

        // --------------------------------------------------------
        // 内层循环: 为当前 WorkKind 授权多个请求
        // --------------------------------------------------------
        for {
            // 检查是否有等待请求
            hasWaiting, demuxHandle := granter.requesterHasWaitingRequests()
            if !hasWaiting {
                break  // 没有等待请求，跳到下一个 WorkKind
            }

            // 尝试获取资源
            res := granter.tryGetLocked(1, demuxHandle)

            switch res {
            case grantSuccess:
                // ----------------------------------------
                // 授权成功分支
                // ----------------------------------------

                // 调用 requester.granted() 通知等待的 goroutine
                tookCount := requester.granted(coord.mu.grantChainID)

                // 更新 granter 的使用量
                granter.tookWithoutPermissionLocked(tookCount, demuxHandle)

                // 检查是否达到 burst 限制
                grantBurstCount++
                if grantBurstCount == grantBurstLimit {
                    // 达到限制，激活 Grant Chain 并返回
                    coord.mu.grantChainActive = true
                    if startingChain {
                        coord.mu.grantChainStartTime = timeutil.Now()
                    }
                    return  // ← 等待 continueGrantChain() 调用
                }
                // 继续授权当前 WorkKind 的下一个请求

            case grantFailDueToSharedResource:
                // ----------------------------------------
                // CPU 过载分支
                // ----------------------------------------

                // 共享资源（CPU）不足，停止所有授权
                break OuterLoop

            case grantFailLocal:
                // ----------------------------------------
                // 本地资源不足分支
                // ----------------------------------------

                // 当前 WorkKind 的 slots/tokens 耗尽
                // 跳到下一个 WorkKind
                break
            }
        }
    }

    // ============================================================
    // 阶段 4: 授权完成，清理 Grant Chain 状态
    // ============================================================

    // 所有 WorkKind 都授权完成，或没有更多资源
    coord.mu.grantChainID++  // 增加 chain ID，使旧 chain 失效
    coord.mu.grantChainActive = false
    coord.mu.grantChainStartTime = time.Time{}
}
```

#### 4.1.2 WorkKind 优先级顺序

```go
// 源代码位置: pkg/util/admission/admission.go:475-510

const (
    KVWork              WorkKind = 0  // 最高优先级
    SQLKVResponseWork   WorkKind = 1  // 中等优先级
    SQLSQLResponseWork  WorkKind = 2  // 最低优先级
    numWorkKinds        int = 3
)
```

**优先级原理**：

```
为什么 KVWork 优先级最高？

1. 系统稳定性：
   ├─► KVWork 是 Raft 复制日志的处理
   ├─► Raft 需要及时响应以维持集群健康
   └─► 延迟会导致 Leader 选举超时

2. 资源利用率：
   ├─► KVWork 完成后会释放锁和资源
   ├─► SQL 层的工作才能继续进行
   └─► 阻塞 KVWork 会导致整个系统停滞

3. 延迟敏感度：
   ├─► KVWork: P99 < 10ms
   ├─► SQLKVResponseWork: P99 < 100ms
   └─► SQLSQLResponseWork: P99 < 500ms

示例：假设 Grant Chain 正在运行

Grant Chain 遍历顺序：
    ↓
KVWork (index=0)
    ├─► 授权 2 个 KVWork 请求
    ├─► grantBurstCount = 2
    └─► 达到 burst 限制，返回
    ↓
(等待 continueGrantChain 调用)
    ↓
SQLKVResponseWork (index=1)
    ├─► 授权 2 个 SQLKVResponseWork 请求
    ├─► grantBurstCount = 2
    └─► 达到 burst 限制，返回
    ↓
(等待 continueGrantChain 调用)
    ↓
SQLSQLResponseWork (index=2)
    ├─► 授权 2 个 SQLSQLResponseWork 请求
    ├─► grantBurstCount = 2
    └─► 达到 burst 限制，返回 或 所有完成
```

### 4.2 Grant Chain 机制详解

#### 4.2.1 Grant Chain 的三个阶段

**阶段 1: 启动 (Start)**

```go
// 源代码位置: grant_coordinator.go:761-768
startingChain := false
if coord.useGrantChains && !coord.mu.grantChainActive {
    coord.mu.grantChainID++       // 例如: 1 → 2
    coord.mu.grantChainIndex = 0  // 从 KVWork 开始
    startingChain = true
}

// 第一次授权
grantBurstCount := 0
for ... {
    if grantBurstCount == grantBurstLimit {  // 例如: 2
        coord.mu.grantChainActive = true  // ← 激活 Grant Chain
        coord.mu.grantChainStartTime = timeutil.Now()
        return  // ← 等待第一个被授权的 goroutine 调用 continueGrantChain
    }
}
```

**阶段 2: 继续 (Continue)**

```go
// 源代码位置: grant_coordinator.go:591-617
func (coord *GrantCoordinator) continueGrantChain(_ WorkKind, grantChainID grantChainID) {
    // 检查 chain ID 是否匹配
    if grantChainID == noGrantChain {
        return
    }

    coord.mu.Lock()
    defer coord.mu.Unlock()

    // Chain ID 不匹配，说明已被终止
    if grantChainID != coord.mu.grantChainID {
        return  // ← Chain 已失效，退出
    }

    // Chain ID 匹配，继续授权
    if coord.mu.grantChainActive {
        coord.tryGrantLocked()  // ← 继续下一批授权
    }
}
```

**调用路径**：
```
Goroutine A (被授权):
    ↓
channel 收到通知 (chainID = 2)
    ↓
q.granter.continueGrantChain(2)
    ↓
slotGranter.continueGrantChain(2)
    ↓
coord.continueGrantChain(KVWork, 2)
    ↓
检查 chainID == coord.mu.grantChainID?
    ├─► Yes → tryGrantLocked()  ← 继续授权
    └─► No → return             ← Chain 已终止
```

**阶段 3: 终止 (Terminate)**

```go
// 源代码位置: grant_coordinator.go:618-651
func (coord *GrantCoordinator) tryTerminateGrantChain() bool {
    // 条件 1: Grant Chain 必须是激活状态
    if !coord.mu.grantChainActive {
        return false
    }

    // 条件 2: 检查 Grant Chain 持续时间
    // 如果 Chain 刚启动（< 1ms），不终止
    // 原因：避免频繁的 Chain 启动/终止
    if timeutil.Since(coord.mu.grantChainStartTime) < time.Millisecond {
        return false
    }

    // 终止 Grant Chain
    coord.mu.grantChainID++         // 增加 ID，使旧 Chain 失效
    coord.mu.grantChainActive = false
    coord.mu.grantChainStartTime = time.Time{}

    return true  // ← 终止成功
}
```

**终止触发条件**：
1. **高优先级请求到达**：
   ```go
   // grant_coordinator.go:485-487
   if coord.mu.grantChainActive && coord.mu.grantChainIndex >= workKind {
       coord.tryTerminateGrantChain()  // 终止当前 Chain
   }
   ```
   - 例如：Chain 正在处理 SQLSQLResponseWork（index=2）
   - 突然有 KVWork（index=0）请求归还资源
   - 终止 Chain，重新从 KVWork 开始授权

2. **所有请求都已授权**：
   ```go
   // grant_coordinator.go:892-895
   coord.mu.grantChainID++
   coord.mu.grantChainActive = false
   ```
   - 遍历完所有 WorkKind，没有更多等待请求

3. **CPU 过载**：
   ```go
   // grant_coordinator.go:754-757
   if coord.mu.cpuOverloadIndicator.isOverloaded() {
       return  // Grant Chain 自动失效
   }
   ```

#### 4.2.2 Grant Chain 完整示例

**场景设定**：
- 系统负载：中等（50% CPU）
- 等待队列：
  - KVWork: 5 个请求
  - SQLKVResponseWork: 3 个请求
  - SQLSQLResponseWork: 2 个请求

**执行时间线**：

```
T0: returnGrant() 被调用（某个请求完成）
    ↓
    调用 tryGrantLocked()

T1: 启动 Grant Chain
    ├─► grantChainID = 100
    ├─► grantChainIndex = 0 (KVWork)
    └─► grantChainActive = false

T2: 为 KVWork 授权第 1 个请求
    ├─► granter.tryGetLocked() = grantSuccess
    ├─► requester.granted(100) → 通知 Goroutine A
    └─► grantBurstCount = 1

T3: 为 KVWork 授权第 2 个请求
    ├─► granter.tryGetLocked() = grantSuccess
    ├─► requester.granted(100) → 通知 Goroutine B
    └─► grantBurstCount = 2

T4: 达到 burst 限制
    ├─► grantChainActive = true  ← 激活 Grant Chain
    ├─► grantChainStartTime = T4
    └─► return  ← 退出，等待 continueGrantChain

--- 时间间隔：等待 Goroutine 调度 ---

T5: Goroutine A 被调度器唤醒
    ├─► 从 channel 收到 chainID = 100
    ├─► 执行业务逻辑（省略）
    └─► 调用 continueGrantChain(100)

T6: continueGrantChain(100) 执行
    ├─► 检查 chainID == 100 ✓
    ├─► 调用 tryGrantLocked()
    └─► grantChainIndex 仍为 0 (KVWork)

T7: 继续为 KVWork 授权第 3 个请求
    ├─► granter.tryGetLocked() = grantSuccess
    ├─► requester.granted(100) → 通知 Goroutine C
    └─► grantBurstCount = 1 (重置)

T8: 为 KVWork 授权第 4 个请求
    ├─► granter.tryGetLocked() = grantSuccess
    ├─► requester.granted(100) → 通知 Goroutine D
    └─► grantBurstCount = 2

T9: 达到 burst 限制，再次返回
    └─► return  ← 等待下一次 continueGrantChain

--- 时间间隔 ---

T10: Goroutine B 或 C 调用 continueGrantChain(100)
     ↓
     继续授权 KVWork 第 5 个请求
     ↓
     KVWork 队列清空

T11: grantChainIndex 递增到 1 (SQLKVResponseWork)
     ↓
     为 SQLKVResponseWork 授权 2 个请求
     ↓
     达到 burst 限制，返回

T12: SQLKVResponseWork 队列清空
     ↓
     grantChainIndex 递增到 2 (SQLSQLResponseWork)
     ↓
     为 SQLSQLResponseWork 授权 2 个请求
     ↓
     所有队列清空

T13: 所有 WorkKind 遍历完成
     ├─► grantChainID = 101  ← 增加 ID
     ├─► grantChainActive = false  ← 停用
     └─► Grant Chain 结束

总共授权: 5 + 3 + 2 = 10 个请求
Chain 持续时间: T13 - T4 ≈ 数毫秒到数十毫秒
```

**性能优势**：

传统方式（每次授权一个请求）：
```
10 个请求 × 每次锁竞争 1μs = 10μs
10 个请求 × 每次上下文切换 10μs = 100μs
总计: ~110μs
```

Grant Chain 方式（批量授权）：
```
5 次批量授权 × 每次锁竞争 1μs = 5μs
10 个请求 × 每次上下文切换 10μs = 100μs
总计: ~105μs + 自然节流效果
```

**关键收益**：
- 减少锁竞争次数：从 10 次降到 5 次
- 自然节流：利用 goroutine 调度器的调度延迟
- CPU 感知：通过 goroutine 运行情况间接感知 CPU 负载

### 4.3 granted() 函数：通知等待的 Goroutine

**源代码位置**: `pkg/util/admission/work_queue.go:1140-1222`

```go
// granted 被 GrantCoordinator 调用，用于通知一个等待的请求已被授权
// 返回值：实际使用的 slots/tokens 数量
func (q *WorkQueue) granted(grantChainID grantChainID) int64 {
    now := q.timeNow()

    q.mu.Lock()

    // 队列为空检查
    if len(q.mu.tenantHeap) == 0 {
        q.mu.Unlock()
        return 0
    }

    // ============================================================
    // 步骤 1: 选择要授权的租户
    // ============================================================

    // tenantHeap 顶部是 used/weight 最小的租户（最公平）
    tenant := q.mu.tenantHeap[0]

    var item *waitingWork

    // ============================================================
    // 步骤 2: 从租户的等待队列中取出请求
    // ============================================================

    // 优先从 waitingWorkHeap 取（已就绪的请求）
    if len(tenant.waitingWorkHeap) > 0 {
        item = heap.Pop(&tenant.waitingWorkHeap).(*waitingWork)
    } else {
        // waitingWorkHeap 为空，从 openEpochsHeap 取
        // （系统空闲时才会发生）
        item = heap.Pop(&tenant.openEpochsHeap).(*waitingWork)
    }

    // ============================================================
    // 步骤 3: 更新统计信息
    // ============================================================

    // 记录等待时延
    waitDur := now.Sub(item.enqueueingTime)
    tenant.priorityStates.updateDelayLocked(item.priority, waitDur, false)

    // 更新租户使用量
    tenant.used += uint64(item.requestedCount)

    // 更新租户在 tenantHeap 中的位置
    if isInTenantHeap(tenant) {
        q.mu.tenantHeap.fix(tenant)  // 重新堆化
    } else {
        q.mu.tenantHeap.remove(tenant)  // 移除（所有请求已处理）
    }

    requestedCount := item.requestedCount
    tenantID := tenant.id

    q.mu.Unlock()

    // ============================================================
    // 步骤 4: 通知等待的 Goroutine
    // ============================================================

    if !item.replicated.Enabled {
        // 普通请求：通过 channel 通知
        item.ch <- grantChainID  // ← 核心！发送通知

        // Goroutine 会在 Admit() 的 select 中收到这个值
        // 然后调用 continueGrantChain(grantChainID)

    } else {
        // 复制写入：直接调用回调
        // （Store 层不使用 Grant Chain）
        defer releaseWaitingWork(item)

        q.onAdmittedReplicatedWork.admittedReplicatedWork(
            roachpb.MustMakeTenantID(tenantID),
            item.priority,
            item.replicated,
            item.requestedCount,
            item.createTime,
            true,  // coordMuLocked
        )

        q.metrics.incAdmitted(item.priority)
        q.metrics.recordFinishWait(item.priority, waitDur)
    }

    return requestedCount
}
```

### 5.1 AdmittedWorkDone() 资源归还

**源代码位置**: `pkg/util/admission/work_queue.go:1224-1254`

```go
// AdmittedWorkDone 在工作完成后被调用，用于归还资源
// 参数:
//   - tenantID: 租户 ID
//   - cpuTime: 实际消耗的 CPU 时间（仅 SQL work 使用）
func (q *WorkQueue) AdmittedWorkDone(
    tenantID roachpb.TenantID,
    cpuTime time.Duration,
) {
    tid := tenantID.ToUint64()

    q.mu.Lock()
    defer q.mu.Unlock()

    // ============================================================
    // 步骤 1: 更新租户统计
    // ============================================================

    tenant, ok := q.mu.tenants[tid]
    if !ok {
        // 租户不存在（可能已被 GC）
        // 直接归还资源到 granter
        q.granter.returnGrant(1, 0 /* demuxHandle */)
        return
    }

    // ============================================================
    // 步骤 2: 减少租户使用量
    // ============================================================

    var returnCount int64

    if q.workKind == KVWork {
        // KVWork: 归还 1 slot
        returnCount = 1
        if tenant.used >= 1 {
            tenant.used -= 1
        } else {
            tenant.used = 0
        }

    } else {
        // SQL Work: 根据实际 CPU 时间计算 tokens
        // tokens = cpuTime / slotDuration
        // 例如: cpuTime=100ms, slotDuration=10ms → 10 tokens
        returnCount = int64(cpuTime / q.slotDuration)
        if returnCount <= 0 {
            returnCount = 1  // 最少归还 1 token
        }

        if tenant.used >= uint64(returnCount) {
            tenant.used -= uint64(returnCount)
        } else {
            tenant.used = 0
        }
    }

    // ============================================================
    // 步骤 3: 更新 tenantHeap
    // ============================================================

    if isInTenantHeap(tenant) {
        // 租户仍在 heap 中（有等待请求）
        q.mu.tenantHeap.fix(tenant)  // 重新堆化（used 减少了）
    }

    // ============================================================
    // 步骤 4: 归还资源到 Granter
    // ============================================================

    // 这里不需要持有 q.mu 锁
    // returnGrant 会获取 GrantCoordinator.mu 锁
    q.mu.Unlock()

    q.granter.returnGrant(returnCount, 0 /* demuxHandle */)

    q.mu.Lock()
    // (函数结束时再次释放 q.mu)
}
```

**returnGrant() 触发授权**：

```go
// 源代码位置: grant_coordinator.go:523-547
func (coord *GrantCoordinator) returnGrant(
    workKind WorkKind,
    count int64,
    demuxHandle int8,
) {
    coord.mu.Lock()
    defer coord.mu.Unlock()

    // 归还资源到 granter
    coord.granters[workKind].returnGrantLocked(count, demuxHandle)

    // ============================================================
    // Grant Chain 处理逻辑
    // ============================================================

    if coord.mu.grantChainActive {
        // Grant Chain 正在运行

        if coord.mu.grantChainIndex > workKind &&
            coord.granters[workKind].requesterHasWaitingRequests() {
            // 情况：Chain 正在处理低优先级 WorkKind
            //      但高优先级 WorkKind 有等待请求

            // 终止当前 Chain
            if !coord.tryTerminateGrantChain() {
                return  // 终止失败，等待 Chain 自然结束
            }
            // 继续执行 tryGrantLocked()

        } else {
            // 情况：Chain 会处理这个 WorkKind，或没有等待请求
            return  // 不做任何事，让 Chain 继续运行
        }
    }

    // 没有 Grant Chain，或 Chain 已终止
    // 尝试为等待的请求授权
    coord.tryGrantLocked()
}
```

```go
// grant_coordinator.go:749-898

func (coord *GrantCoordinator) tryGrantLocked() {
    // CPU 检查
    if coord.mu.cpuOverloadIndicator.isOverloaded() {
        return  // CPU 过载，暂停授权
    }
    // 假设 CPU 正常 ✓

    // 启动 Grant Chain
    coord.mu.grantChainID++  // 例如: 42 → 43
    coord.mu.grantChainIndex = 0  // 从 KVWork 开始

    grantBurstCount := 0

    // 遍历 KVWork
    for {
        // 检查是否有等待请求
        hasWaiting, _ := coord.granters[KVWork].requesterHasWaitingRequests()
        // hasWaiting = true ✓

        // 尝试获取 slot
        res := coord.granters[KVWork].tryGetLocked(1, 0)
        // usedSlots=99 < totalSlots=100 ✓
        // res = grantSuccess

        // 授权成功
        tookCount := coord.queues[KVWork].granted(43)  // ← 调用 granted
        // tookCount = 1

        coord.granters[KVWork].tookWithoutPermissionLocked(1, 0)
```

**kvStoreTokenGranter 的三阶段检查**：

```go
// 源代码位置: pkg/util/admission/granter.go:428-565

type kvStoreTokenGranter struct {
    coord                *GrantCoordinator
    regularTokensMetric  *metric.Gauge
    elasticTokensMetric  *metric.Gauge

    // IO token buckets
    ioTokens         int64
    l0Tokens         int64
    diskBWTokens     int64
}

func (sg *kvStoreTokenGranter) tryGet(
    wc admissionpb.WorkClass,
    info StoreWorkInfo,
) bool {
    requested := info.WriteBytes

    if wc == admissionpb.RegularWorkClass {
        // ====================================================
        // Regular Work: 单阶段检查
        // ====================================================

        // 只检查 IO tokens
        if sg.ioTokens >= requested {
            sg.ioTokens -= requested
            sg.regularTokensMetric.Dec(requested)
            return true  // ✓ 授权成功
        }
        return false  // ✗ IO tokens 不足

    } else {
        // ====================================================
        // Elastic Work: 三阶段检查
        // ====================================================

        // 阶段 1: L0 压力检查
        if sg.l0Tokens <= 0 {
            return false  // ✗ L0 压力过大
        }

        // 阶段 2: 磁盘带宽检查
        if sg.diskBWTokens <= 0 {
            return false  // ✗ 磁盘带宽饱和
        }

        // 阶段 3: IO tokens 检查
        if sg.ioTokens >= requested {
            sg.ioTokens -= requested
            sg.l0Tokens -= requested
            sg.diskBWTokens -= requested
            sg.elasticTokensMetric.Dec(requested)
            return true  // ✓ 授权成功
        }

        return false  // ✗ IO tokens 不足
    }
}
```

**Regular vs Elastic 授权策略对比**：

| 检查项       | Regular Work | Elastic Work | 说明                    |
|-------------|--------------|--------------|------------------------|
| IO Tokens   | ✓            | ✓            | 基础 IO 容量检查        |
| L0 Tokens   | ✗            | ✓            | L0 压力检查（防止 LSM 崩溃）|
| Disk BW     | ✗            | ✓            | 磁盘带宽检查（防止饱和） |

**为什么 Regular 不检查 L0 和 Disk BW？**

```
设计原则：

Regular Work（用户请求）：
├─► 优先级最高
├─► 必须尽快处理，避免用户感知延迟
└─► 允许短暂的 L0 压力或磁盘饱和

Elastic Work（后台任务，如 Compaction）：
├─► 优先级较低
├─► 可以延迟处理
└─► 严格限制，避免影响 Regular Work

示例场景：

L0 sublevel count = 18（接近上限 20）

Regular Write:
├─► 检查 IO tokens ✓ (足够)
├─► 跳过 L0 检查
└─► 授权成功 → 可能导致 L0 = 19

Elastic Write (Compaction):
├─► 检查 L0 tokens ✗ (l0Tokens = -100)
└─► 拒绝 → 等待 L0 降低

结果：
├─► Regular 继续服务用户请求 ✓
└─► Elastic 暂停，等待系统恢复 ✓
```

## 总结

### 生命周期关键点回顾

1. **准入请求 (Admit)**：
   - 快速路径：队列为空 + 资源充足 → 1-10μs
   - 慢速路径：排队等待 → 1ms-数秒

2. **资源授权 (Grant)**：
   - Grant Chain：批量授权，减少锁竞争
   - 优先级顺序：KVWork > SQLKVResponseWork > SQLSQLResponseWork
   - 自然节流：利用 goroutine 调度延迟

3. **工作执行 (Execute)**：
   - KVWork: Raft 提议
   - SQL Work: KV/SQL 响应处理

4. **资源释放 (Done)**：
   - AdmittedWorkDone: 归还 slots/tokens
   - returnGrant: 触发下一轮授权
   - Grant Chain 继续或终止

### 性能关键指标

| 指标                  | 快速路径  | 慢速路径（无竞争） | 慢速路径（高竞争） |
|----------------------|----------|-------------------|-------------------|
| Admit 延迟           | 1-10μs   | 1-10ms            | 10ms-数秒         |
| 锁竞争次数           | 1次       | 2次                | 多次              |
| Goroutine 切换       | 0次       | 1次                | 1次               |
| Channel 通信         | 0次       | 1次                | 1次               |
| Grant Chain 长度     | N/A      | 2-10个请求        | 2-10个请求        |
